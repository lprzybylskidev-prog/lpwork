# AGENTS.md

## Scope

These instructions govern LPWork framework development. They apply to framework code under `LPWork`, framework-owned resources, framework tests, framework CLI commands, the development workspace, and the default skeleton/demo application only when it is being used as a framework integration surface.

These instructions do not define how future end-user application code should be designed inside `App`. Application-specific coding rules should live in dedicated skills or project-level guidance later. Until then, treat `App` as framework-facing composition, a skeleton, or a test/demo surface, not as the source of framework internals.

## Project Purpose

LPWork is a small PHP framework learning project. Keep the code easy to understand, easy to test, and consistent with the existing framework style. Prefer small, complete steps that leave the framework in a better architectural state than before.

The main goal is to avoid large corrective refactors later. When adding or changing behavior, make the design decision at the point where complexity appears instead of postponing it to a future cleanup.

## Framework Development Principles

- Before implementing a non-trivial change, identify the responsibilities involved: orchestration, parsing, resolution, dispatching, configuration, state storage, side effects, observability, and error handling.
- Design each change as close as practical to its intended final architecture. When adding a reusable framework capability, also check the touched area for existing code that should immediately use it, and wire it in during the same change when it improves clarity without expanding scope into an unrelated subsystem.
- A class should have one primary reason to change. If a change adds a second responsibility to an existing class, introduce a small collaborator instead of letting the class grow.
- Keep orchestration separate from detailed work. Orchestrators coordinate the flow; they should not also parse input, resolve dependencies, validate implementation details, or perform side effects directly.
- Keep data models focused on representing state and behavior around that state. Do not make them responsible for discovering, parsing, or loading their own external input unless they are explicit boundary objects.
- Introduce abstractions when they reduce real complexity, isolate a responsibility, or match an existing project pattern. Do not add abstractions only to anticipate unknown future features.
- When similar logic appears for the second time, look for a shared role, resolver, factory, or support helper before duplicating the pattern.
- If a method needs a long sequence of unrelated steps, split those steps into named collaborators or private methods with clear boundaries.

## Framework, Skeleton, And Application Boundaries

- Keep framework mechanisms in the framework namespace and application composition in the application namespace.
- Framework internals must not assume application file names, directory structure, module layout, or concrete application classes except through explicit framework-owned contracts, provider APIs, configuration, or skeleton generation boundaries.
- The default `App` tree may be used as a skeleton/demo/integration surface, but framework behavior belongs in `LPWork`.
- Framework-owned generators may create or update application provider declarations, but the shared mechanics for those declarations belong in framework code.
- Application-owned loadable files should be exposed through explicit provider declaration methods. Framework code should consume those declarations rather than scanning or guessing application structure.
- Application provider file paths should be project-root-relative strings. Where path resolution is needed, use the application path abstraction instead of manual path concatenation in application providers.
- Entrypoints should stay thin: bootstrap, create the relevant kernel, and hand off execution.
- Code needed after installation must either belong to the framework, be generated by framework tooling, or be part of the intentional skeleton surface.

## Modules, Providers, And Registration

- Framework modules should own their service registration, commands, routes, migrations, health checks, config definitions, translations, diagnostics, and resources through focused providers.
- Service providers should register services and declarations. They should not perform runtime side effects, execute commands, open network connections, mutate storage, or do heavy work during registration.
- If provider registration grows across independent responsibilities, split it into focused providers, registrars, factories, or definition objects.
- A provider may coordinate registration, but detailed parsing, validation, construction variants, and discovery should live in dedicated collaborators.
- Framework-owned persistent support tables, such as queue, cache, session, scheduler, locks, notifications, or other runtime storage tables, must be declared as framework migrations under `LPWork` and registered by the owning framework module. Do not create those tables lazily from repositories, drivers, workers, commands, or other runtime side-effect paths. The migration repository may still own its own bootstrap storage needed to track migration history.
- When adding a new framework module or capability, update the module catalog, module health coverage, and user-facing framework metadata when those surfaces are meant to represent available framework capabilities.

## Public API Shape

- Keep public APIs small and intentional. Add public methods only for behavior needed by the current feature.
- A public method should express one action or query. Avoid methods that behave as both getter and setter or change meaning based on optional arguments.
- Avoid broad union return types for fluent APIs, state access, and mutation. If a method would need to return unrelated types, split the API into clearer operations.
- Prefer explicit named constructors for boundary creation and environment-specific setup.
- Prefer immutable objects when practical. Use `readonly` classes or readonly promoted properties for immutable state.
- Keep constructors simple. Inject collaborators through constructor arguments and avoid doing heavy work in constructors.
- Avoid magic behavior. Make lifecycle, dependencies, and side effects explicit.

## Framework Documentation

- Public framework classes, contracts, enums, and public methods in `LPWork` should have docblocks that explain the intent of the API, not just restate its type signature.
- Private and protected methods should have docblocks only when the behavior, format, lifecycle, side effect, or algorithm is not obvious from the name and surrounding code.
- Use `@param` descriptions when they add meaning beyond strict PHP types, such as accepted formats, allowed values, units, lifecycle expectations, side effects, array shapes, or generic collection contents.
- Do not add empty PHPDoc that only duplicates native types, such as `@param string $name` without explaining what the string represents.
- User-facing configuration files under `App/Shared/Configs` should document what each option controls, available values, supported drivers, required and optional driver configuration, defaults, examples, and external service requirements.
- Keep inline config comments useful but concise. Move longer explanations, recipes, and cross-feature guidance into `docs/`.
- Documentation under `docs/` should explain how to build applications with LPWork: configuration, modules, routing, controllers, requests, responses, views/assets, validation, persistence, cache, session, locks, queues, scheduler, mail, notifications, broadcasting, security, storage, testing, CLI commands, health/about, and development tooling.
- Primary user documentation should focus on the framework capabilities exposed to applications. Avoid internal architecture explanations unless they are necessary to use the framework correctly.
- The framework repository `README.md`, `lpwork about` output, and default welcome page capability messaging must stay aligned. When one surface presents LPWork capabilities, design goals, or project status differently, update the other relevant surfaces in the same change so public repository documentation, CLI orientation, and the default demo experience do not drift.
- When adding or changing a user-facing framework capability, update the relevant PHPDoc, config comments, and `docs/` pages in the same change when those surfaces are affected.

## Configuration, Drivers, Managers, Resolvers, And Factories

- Keep parsing separate from loading. Loading finds the source; parsing turns source content into structured data.
- Keep configuration readers focused on reading and validating configuration values, not on constructing unrelated runtime services.
- Prefer structured data and explicit validation over ad hoc string manipulation.
- When parsing rules become non-trivial, move them into a dedicated parser covered by focused tests.
- Containers resolve dependencies; they should expose clear binding and resolution behavior without duplicating registration paths.
- Dependency resolution should carry its own context when multiple resolution steps need shared state.
- Managers should coordinate access to configured services. They should not grow indefinitely by also parsing config, constructing drivers, validating every option, and caching instances inline.
- Use factories for constructing variants when more than one variant or strategy exists.
- Use resolvers for turning declarations, names, paths, or metadata into concrete runtime objects.
- Validate unsupported variants explicitly and fail with domain-specific exceptions.

## Runtime Flow: HTTP, CLI, Queue, And Schedule

- Keep the flow explicit and consistent across runtime modes: request/input, orchestration, response/output, emitter or worker side effect.
- Kernels orchestrate request creation, response creation, exception handling, middleware flow, dispatching, and emitter execution.
- Kernels should delegate detailed work to focused collaborators once the logic is more than a trivial call.
- Request objects wrap input data and expose request state. They should not perform application dispatch, dependency resolution, or output.
- Response objects represent status, headers, cookies, and body data. They should not emit themselves.
- Emitters are responsible for side effects such as writing output, sending headers, setting status codes, or writing to streams.
- Workers, schedulers, and long-running processes should make lifecycle, error handling, retry/exit behavior, and observable progress explicit.

## Routing, Middleware, And Dispatch

- Keep route matching, middleware resolution, middleware execution, and controller/action dispatch as separate roles.
- Use middleware for cross-cutting request or command concerns when the behavior naturally wraps execution, such as authentication, authorization, throttling, production safety, logging, tracing, maintenance mode, input normalization, or similar pre/post execution policy.
- Do not bury reusable cross-cutting policy inside controllers, route closures, command handlers, dispatchers, workers, schedulers, or kernels when an HTTP or CLI middleware pipeline can express the boundary cleanly.
- Do not use middleware to hide core control flow or make required behavior hard to trace.
- Route objects should describe routes and route metadata. They should not own unrelated dispatch or container logic.
- Middleware declarations should be resolved and validated in one place before the pipeline is executed.
- Dispatching should be responsible for invoking the resolved action and validating its response type.
- Pipeline construction should stay explicit and testable.

## Commands, Project Tasks, And Production Safety

- Built-in commands should be thin adapters around input, output, validation, and a focused collaborator that performs the work.
- Longer command behavior belongs in runners, creators, resolvers, renderers, or managers covered by focused tests.
- When adding or changing built-in framework commands, global CLI options, generator workflows, frontend/browser tooling commands, database/cache/runtime workflows, or other user-facing console capabilities, update the global `lpwork --help` output and its tests in the same change.
- Command metadata, help text, completion behavior, and tests should change with the command contract.
- Commands that wrap external tools should preserve meaningful exit codes and make the underlying task clear.
- CLI commands that mutate secrets, configuration, schema, stored data, cache state, or other persistent runtime state must make the side effect explicit in their public command contract.
- Commands that are destructive or production-sensitive, such as app key generation, key rotation, database reset, `migrate:fresh`, cache clearing with data loss, or similar operations, must refuse to run in production unless the user passes `--force`.
- Commands that would overwrite an existing secret or persistent value must fail by default and only overwrite it when `--force` is present.
- Production detection must go through the framework configuration/environment boundary, not ad hoc reads from globals or process state inside command handlers.
- The `--force` check should live in a shared CLI safety boundary, preferably middleware or a reusable command collaborator, once more than one command needs it.
- Tests for production-sensitive commands must cover safe refusal in production, successful execution with `--force`, and normal development behavior.

## Observability, Health, About, And Debugging

- When adding or changing framework behavior, consider whether it should be observable through logging, the shared debug context, health checks, `about`, the debugbar, or the debug exception page before finalizing the design.
- Do not ship placeholder diagnostics when the framework can measure the real lifecycle data. If an observable action has a meaningful start and finish, record its actual duration instead of a generic count or synthetic marker.
- When adding a new framework module, runtime dependency, driver, cache, service, devcontainer service, or other framework capability that can meaningfully be checked at runtime, add or update its `health:check` coverage in the same change.
- When adding a new framework capability, runtime dependency, driver, cache, service, devcontainer service, or other framework metadata that is useful for diagnostics or project orientation, add or update the `about` command output in the same change.
- When new framework behavior adds meaningful runtime diagnostics, add the useful data to the debugbar in the same change. If the same data helps exception triage, add it to the debug exception page through the shared debug context or diagnostics snapshot in the same change.
- Prefer focused collectors, reporters, middleware, or listeners over scattering logging and debug-context mutation through orchestration code.
- Do not log sensitive values by default; follow the existing `APP_DEBUG` boundary for secrets, bindings, payloads, tokens, email addresses, and other user data.
- Avoid noisy default logging for high-volume internal activity unless it is explicitly configurable or clearly useful for diagnosing failures.
- Exceptions should still flow through the existing error-handling boundary; observability hooks should add context, not replace exception handling.

## Events And Extensibility

- When a feature creates a meaningful framework or application lifecycle event, consider whether optional reactions should be expressed through events and listeners instead of direct dependencies or inline side effects.
- Use events/listeners for extensible reactions such as lifecycle notifications, audit/logging hooks, metrics, mail/notification triggers, queue handoff, cache invalidation, or other optional follow-up work.
- Do not use events/listeners to hide core control flow, replace middleware, replace explicit service calls needed for the main result, or make required behavior hard to trace.
- Event payloads should be explicit value objects or focused event classes, not loosely shaped arrays.

## Globals, Static State, And Lifecycle

- Access PHP globals only at explicit framework boundaries.
- Superglobals, process input, output functions, streams, sessions, and headers must be isolated behind boundary objects or boundary factories.
- Domain and framework internals must not read superglobals, emit headers, write directly to output streams, or call session functions.
- Any static or global state must have an explicit lifecycle. It must be resettable for tests, safe for repeated bootstrap in one process, or replaced by an instance registered in the container.
- Tests should never need reflection to reset framework state. If they do, add a proper lifecycle boundary.
- Avoid hidden mutation of global state from orchestration code. Configure shared state in bootstrap or provider layers where lifecycle is visible.

## Framework Helpers

- Helper logic belongs in the framework module that owns the behavior, such as HTTP, routing, URL generation, security, session, view, validation, or console.
- When adding a public convenience helper, create it in its semantic module first and cover it with focused tests there.
- Add a shortcut for public convenience helpers to the static `LPWork\Support\Helpers` aggregate only when the helper is intentionally part of the public convenience surface.
- Keep `Helpers` as a thin delegating catalog only. Do not move parsing, validation, state mutation, dependency resolution, or side effects into the aggregate.
- Do not add global helper functions. Prefer explicit static helper classes with resettable lifecycle where static state is unavoidable.

## Exceptions

- Do not throw base exceptions directly from framework code.
- Create custom exception classes in the nearest domain-specific exception namespace.
- Custom exceptions may extend SPL/base exception classes internally.
- Exception names should describe the failure, not the implementation detail.
- Use named constructors when one exception type represents multiple related failures.
- Error messages should be specific enough to diagnose the failing boundary, missing configuration, invalid declaration, unsupported variant, or unsafe operation.

## Framework-Owned UI, Assets, And Translations

- Framework-owned views, including the welcome page, debugbar, maintenance page, production error page, debug exception page, health pages, and default framework mail template, are backend-rendered PHP/HTML views with shared framework CSS and small vanilla JavaScript only.
- Do not introduce a frontend application stack, bundler, UI framework, icon package, CSS framework, or runtime JavaScript dependency for framework-owned views unless the user explicitly changes the product direction.
- Node, npm, Vite, Vitest, TypeScript, linters, formatters, and Playwright are development tools in this project. Use them to inspect and verify framework UI and frontend tooling, not to turn framework-owned PHP views into a compiled frontend application.
- Framework-owned CSS, JavaScript, views, templates, and translation files should live in the module that owns the behavior or in a shared framework resource location when intentionally reused.
- Avoid large heredoc assets in PHP classes when the asset is a framework-owned resource file.
- When a user-facing framework capability should be visible in the default skeleton/demo experience, update the welcome page capability list and translations in the same change.
- When the user says an existing framework-owned view is only a working/template view, or says to rebuild a view from scratch, treat the old view as content reference only. Preserve the user-facing information it displays, but remove or replace the old layout, visual styling, CSS colors, spacing, markup patterns, and interaction assumptions instead of layering new styling over legacy structure.
- When the user identifies one framework-owned view or component as the visual template for the rest, use that component as the source of truth for palette, surface hierarchy, borders, radius, density, data presentation, empty states, and active/hover states. Do not keep mismatched legacy styles from older iterations.
- Before changing a non-trivial framework UI, inspect the relevant renderer/template and identify the visible states, viewport sizes, interactive controls, overflow risks, and dynamic content that need browser verification.
- After changing a framework UI, run the relevant PHP/Pest tests and use Playwright against the running dev server to capture desktop and mobile screenshots. Inspect the screenshots before reporting completion.
- For debugbar work, capture both the full page and `#lp-debug-bar` where practical. Exercise the relevant tab or interaction before taking the screenshot, such as selecting the Metrics tab when changing metrics UI.
- Treat Playwright screenshots as design feedback. If text overlaps, controls are misaligned, spacing hides relationships, timelines are not visually proportional, or mobile/desktop layouts lose readability, fix the UI before finishing.
- Keep generated Playwright artifacts under `storage/playwright/` or another ignored local diagnostics directory.

## Development Workspace And Devcontainer

- The devcontainer is an LPWork development workspace for VS Code, not a production container or deployment target.
- When a framework feature requires runtime services, PHP extensions, CLI tools, ports, volumes, environment variables, browser tooling, database clients, or other local infrastructure, update `.devcontainer` in the same change so the project works both in the current workspace and from a fresh clone.
- Optional infrastructure services may be available in the default devcontainer so Redis, SQL databases, S3/SQS, SMTP, websocket broadcasting, metrics, browser testing, and similar features can be enabled through configuration without rebuilding the workspace.
- Keep long-running workers and schedulers as explicit opt-in compose profiles because they execute background work rather than merely providing development infrastructure.
- Keep framework defaults simple and lightweight even when the devcontainer provides heavier optional services.
- Keep `health:check`, `about`, and global `lpwork --help` aligned when devcontainer-backed tooling or development workflows change.
- When changing devcontainer files, manually apply the equivalent change to the current workspace environment when practical so the active container does not require a rebuild to continue working.
- Do not trigger or recommend a devcontainer rebuild as the first fix for workspace tooling drift. Prefer installing packages, rerunning setup scripts, recreating individual compose services, or providing host-side Docker commands; recommend a rebuild only when the change cannot be applied safely to the current container and explain why.
- Keep `.devcontainer/setup-lpwork-shell.sh` as the VS Code development shell integration for the `lpwork` executable and bash completion unless a better replacement is introduced.

## Git, Commits, Branches, And Releases

- These Git workflow rules apply only after the framework repository has actually been initialized with Git. Do not initialize Git, create remotes, create branches, commit, tag, or push merely because this section exists.
- LPWork `v1.0.0` is the first public/recruiter-readable framework snapshot, not the end of framework development. Do not describe the project as finished or frozen merely because `v1.0.0` is being prepared; future work should continue through explicit roadmap items, versioned branches where needed, and immutable release tags.
- LPWork uses three-part release versions in the form `MAJOR.MINOR.PATCH`, published as immutable tags such as `v1.0.1`.
- Increment `PATCH` for small bug fixes, release-correctness fixes, documentation clarifications, test adjustments, and behavior that restores the documented or intended framework behavior without changing the public API.
- Increment `MINOR` for larger compatible improvement batches, framework hardening rounds, new options, tooling improvements, or cohesive capability upgrades that do not require applications to change existing public API usage.
- Increment `MAJOR` for larger framework stages, new substantial feature sets, public API or application contract changes, installer contract changes, or architectural rebuilds that materially change how applications integrate with LPWork.
- Once Git is active, AI agents are responsible for preparing clean commits and pushes for completed work when the current task includes repository maintenance, release work, or the user asks to commit/push the finished changes.
- Before committing, inspect the worktree and separate unrelated changes into separate commits. Do not create one large commit that mixes independent framework behavior, documentation, tooling, configuration, formatting, and roadmap changes.
- Commit messages must be written in English and use conventional lowercase prefixes such as `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`, `build:`, or `ci:`.
- Commit scopes may be used when they make the affected area clearer, such as `feat(console): add command metadata reporter`.
- Each commit should represent one coherent reason to change and should include its matching tests, docs, config, or metadata updates when those are part of the same behavior.
- Do not commit generated dependencies, local caches, runtime storage, Playwright artifacts, secrets, private roadmap notes, or framework-maintainer-only local files.
- Work on future framework versions should happen on version-appropriate branches so release stabilization, patch work, and new development do not mix.
- Use immutable release tags such as `v1.0.0` for distributed LPWork snapshots. Do not treat moving branches as installable releases.
- The repository may be public for portfolio/CV visibility while the code is not yet open for public use. Keep `LICENSE.md`, `README.md`, installer text, release notes, and GitHub metadata consistent with the current permission model: viewing the code is allowed, but using, copying, modifying, redistributing, or creating derivative works requires explicit permission unless the owner later replaces the license.
- Before pushing, run the verification appropriate to the committed changes and confirm that the branch contains only the intended commits.
- Pushes are part of the agent's responsibility once Git is active and the user has asked for the completed work to be published or the active roadmap/release step explicitly requires it.

## Testing, Static Analysis, And Architecture

- Write Pest tests for every framework behavior change.
- Build or extend test utilities only for framework boundaries that already exist or are introduced by the current change. Do not invent test clients, fakes, principals, assertions, payload contracts, or helper APIs for future modules that the framework has not implemented yet.
- When framework features add behavior that is naturally exercised through an existing or newly introduced runtime lifecycle or shared framework boundary, such as HTTP requests, CLI commands, dependency injection, filesystem/storage, cache, logging, session, security, throttle, configuration, queue, or schedule, build or extend the matching test utilities in the same change and cover the behavior with integration tests through that boundary, not only unit tests.
- Keep tests small, focused, and named after observable behavior.
- Shared test fixtures, streams, temporary files, fake services, fake emitters, and setup utilities belong in the framework-owned test support tree.
- Avoid global helper functions in test files. Prefer support classes with explicit responsibilities.
- When setup logic repeats, move it into test support during the same change.
- Test support code should follow the same basic discipline as production code: clear names, explicit lifecycle, and custom support exceptions where failures are part of the helper behavior.
- Anonymous classes are acceptable for one-off fakes inside a single test, but repeated fake behavior belongs in support classes.
- Tests may use generic throwables only when the tested behavior requires an arbitrary throwable and no domain-specific meaning is involved.
- Keep PHPStan at the configured maximum level passing.
- Respect existing architecture tests and boundary rules.
- Add or update architecture tests and PHPStan rules when introducing a new project-wide boundary rule.
- If a rule needs a shared allowlist or boundary map, keep that source of truth in one reusable support component.
- Treat static-analysis failures as design feedback. Fix the underlying type, lifecycle, or boundary issue instead of suppressing the error.

## File Organization

- Contracts live in `Contracts` subdirectories.
- Domain-specific exceptions live in `Exceptions` subdirectories.
- Enums live in `Enums` subdirectories.
- Framework-owned resources should live with the owning module or in an explicit shared framework resource location.
- Framework-owned tests and test support should mirror framework ownership and production areas.
- Keep application-owned loadable files under the application directory.
- Remove obsolete files as part of the change that makes them unused.
- Remove placeholder files or give them real responsibility. Do not keep empty files only to preserve a directory shape.
- Remove `.gitkeep` files as soon as a directory contains real project files.

## Refactor Prevention Triggers

When any of the following appears during implementation, stop and make the design smaller before continuing:

- A class gains a second independent responsibility.
- A method mixes orchestration with parsing, resolving, dispatching, validation, or side effects.
- A public method needs to return unrelated types.
- A test needs reflection to reset or modify state.
- Similar construction or validation logic appears for a second time.
- Test setup is copied between files.
- A boundary rule needs the same allowlist in more than one place.
- Static/global state is mutated from a place that does not own its lifecycle.
- A placeholder file is kept without real behavior.

## Private Roadmap Workflow

- Always check `lpwork-roadmap` before starting implementation work.
- Do not use LPWork application skills as guidance for framework roadmap implementation. The skills under `.codex/skills/lpwork-application` and `.codex/skills/lpwork-module-development` are installed application guidance for building `App`; framework work must follow this root `AGENTS.md`, `lpwork-roadmap`, and framework-owned docs/tests instead.
- Treat explicit user direction as a hard constraint, not a suggestion. If the user says to ignore an old implementation, rebuild something from scratch, preserve only specific content, or use a named component as the template, follow that instruction directly and remove conflicting legacy code instead of carrying it forward quietly.
- The private `lpwork-roadmap` file has two intentional sections separated by `-----` and is ignored by Git.
- Above the `-----` separator is working space for the currently selected backend roadmap item only when that item needs design notes, assumptions, or a breakdown into smaller implementation tasks.
- Keep the area above the separator empty when there is no active breakdown or design note worth preserving.
- Below the `-----` separator is the canonical backend roadmap backlog: the remaining backend items to finish before the backend side of the framework is considered complete, plus backlog-level assumptions and sequencing notes.
- Do not move, copy, or promote a backlog item from below the separator to above the separator just because it is being worked on.
- Simple backlog items may be implemented directly from the lower list when the user explicitly selects them or asks to continue with them; after implementation and verification, mark the completed lower-list item by changing `[ ]` to `[x]`.
- More complex backlog items should first be discussed or broken down above the separator into concrete Markdown checkbox tasks: `- [ ] Clear action-oriented task description`.
- When a complex backlog item is broken down above the separator, leave the original backlog item below the separator until the whole item is finished; after all active subtasks are implemented and verified, mark the original lower-list backlog item by changing `[ ]` to `[x]` and clear the temporary upper breakdown unless the user wants to keep notes.
- Roadmap items above the separator must describe concrete, actionable work in one sentence. Prefer the style of a precise refactor or implementation task over vague cleanup notes.
- Backlog items below the separator may be broader than active roadmap items, but they should still be specific enough to support later planning and execution.
- After a design or planning conversation that breaks down a roadmap item, write the resulting active tasks above the `-----` separator and add an `Assumptions:` section above the separator with concise bullets capturing the agreed architectural decisions, constraints, and sequencing assumptions so the implementation context is not lost.
- When roadmap planning reveals an important durable framework rule, update `AGENTS.md` in the same planning pass. Do not add every implementation note, temporary workaround, or one-off reminder; `AGENTS.md` should contain stable framework development principles.
- Assumption bullets are not work items. Do not write them as checkboxes, and keep them concrete enough to guide later implementation.
- Mark completed roadmap items above the separator by changing `[ ]` to `[x]`. Do not mark completion by adding suffixes or trailing symbols.
- The agent is responsible for updating roadmap completion state after finishing and verifying a task: mark upper subtasks complete when using an upper breakdown, and mark completed lower backlog items with `[x]` when the selected backlog item is done.
- During implementation work, use backlog items below the `-----` separator only as the selected roadmap scope and context; do not treat the lower list as active checkbox tasks.
- If there are no active roadmap items above the `-----` separator and the user asks to work on a simple named or next backlog item, implement that selected lower-list item directly without first rewriting it above the separator.
- If there are no active roadmap items above the `-----` separator and the user asks to work on a complex next backlog item, first discuss or draft the breakdown above the separator before implementation begins.
- If there are active roadmap items above the `-----` separator and the user has not specified execution mode, ask whether to work item by item or complete all active items in order.
- When working item by item from an upper breakdown, work only on the first unfinished active roadmap item unless the user explicitly says otherwise.
- When asked to complete all active roadmap items above the separator, process them in order from top to bottom.
- Keep each change scoped to the selected lower backlog item, the current upper roadmap item, or the explicitly requested group of roadmap items.
- Do not combine unrelated refactors with feature work.
- Prefer existing project patterns over introducing new ones.
- Before adding a new system module, check whether the project already has a similar module and mirror its structure when the responsibilities match.
- If a new abstraction is needed, keep it minimal, named by responsibility, and covered by tests.
- Do not leave known design debt behind when it is caused by the current change and can be fixed safely in the same step.

## Verification

- After each completed implementation step, run focused tests for the touched behavior.
- Run PHPStan after focused tests when the change affects types, architecture, containers, configuration, providers, or public APIs.
- Before finishing a larger change, run:
  - `php lpwork test`
  - `php lpwork check`
- Run `php lpwork format` when PHP files were added or when imports/signatures changed.
- Before reporting completion, do a short self-review:
  - Does each touched class have one primary responsibility?
  - Are side effects isolated at boundaries?
  - Does any global or static state have an explicit lifecycle?
  - Are repeated test helpers in the framework-owned test support tree?
  - Are obsolete or placeholder files removed or given real responsibility?
