# AGENTS.md

## Scope

These instructions apply to LPWork applications generated from the framework. They are installed as the generated application's root `AGENTS.md`.

Application code belongs under `App`, application documentation belongs under `docs`, and application tests belong under each module's test tree. Framework code under `LPWork` is a dependency boundary and should be changed only when the user explicitly asks for framework development.

## Required Agent Onboarding

Before non-trivial work, read the relevant LPWork application skills:

- Use `.codex/skills/lpwork-application/SKILL.md` for application-wide work, configuration, integrations, runtime services, quality workflow, documentation, and framework/application boundary decisions.
- Use `.codex/skills/lpwork-module-development/SKILL.md` when adding or changing code under `App/Modules/*`.

Those skills are the operational guide for how to navigate LPWork, how to register loadable application pieces, how to test changes, and how to keep application code aligned with framework expectations.

## Application Development Principles

- Prefer extending the application through `App`, Composer or npm dependencies, service providers, configuration, routes, views, commands, migrations, and module code before modifying `LPWork`.
- Keep framework mechanisms in `LPWork` and application composition in `App`.
- Treat `LPWork` as a dependency boundary unless the task is explicitly about framework behavior.
- Keep entrypoints thin: bootstrap the application, create the relevant kernel, and hand off execution.
- Prefer explicit registration over magic discovery. Register routes, providers, config definitions, views, translations, assets, events, commands, migrations, and other loadable pieces through the appropriate provider or configuration boundary.
- Keep orchestration separate from parsing, resolution, validation, dispatch, state storage, and side effects once the behavior stops being trivial.
- Keep side effects visible. Commands or workflows that mutate secrets, schema, cache state, queues, or stored data should say so in their names, help text, docs, or user-facing output.

## Navigation Before Editing

Before choosing an implementation path, inspect the nearby provider, route declarations, config definitions, tests, existing module shape, command help, and relevant docs. Mirror existing LPWork patterns when the responsibility matches.

For CLI behavior, check `php lpwork --help` and `php lpwork <command> --help`. For configuration behavior, inspect `App/Shared/Configs` and the provider that registers those config files. For module behavior, inspect the module's providers and tests first.

## Modules

LPWork applications are organized around `App/Modules/*`. A module may use the default generated layout or a custom internal layout, as long as its loadable pieces remain explicitly registered.

When adding module behavior:

- Put HTTP routes in the module route declarations and register them through the module's route provider.
- Put controllers, request objects, views, assets, translations, commands, migrations, events, listeners, and tests inside the module that owns the behavior.
- Keep module service providers focused on registration. Do not run commands, open network connections, mutate storage, or do heavy runtime work while registering services.
- If registration grows across independent responsibilities, split it into focused providers, registrars, factories, or definition objects.

## Configuration

Application configuration belongs in `App/Shared/Configs`.

- Use configuration files and providers instead of reading globals directly from application internals.
- Keep option names explicit and document important defaults or external service requirements.
- Add new config definitions through the existing application configuration provider pattern.
- Keep secrets in environment values, not committed config files.

## Framework Extension

LPWork intentionally provides basic framework tools rather than a closed ecosystem. Applications may integrate ORM libraries, Twig, Vue/React/Svelte, API tooling, auth packages, cloud SDKs, and other Composer or npm dependencies.

Integrate those dependencies through application providers and configuration. Add framework-level abstractions only when the user wants the framework itself to grow.

## Quality Workflow

Run focused verification first, then broaden when the change touches shared behavior.

- Use `php lpwork test --module=<Name>` for a narrow module backend/frontend change when module filtering applies.
- Use `php lpwork test --backend` or `php lpwork test --frontend` when the changed area is limited to one side.
- Use `php lpwork test` before finishing broader application changes.
- Use `php lpwork check` when PHP types, architecture boundaries, containers, providers, configuration, or public APIs changed.
- Use `php lpwork format` when PHP files were added or when imports, signatures, or formatting changed.
- Use `php lpwork frontend:check` for frontend assets, TypeScript, CSS, linting, or formatting-sensitive frontend work.
- Use browser checks when a UI behavior or view needs visual/runtime verification.
- Use `php lpwork test:lpwork` only when the user asked for framework work or the application change intentionally touches framework code under `LPWork`.

Treat PHPStan/static-analysis failures as design feedback. Fix the type, lifecycle, boundary, or registration issue instead of suppressing it.

## Testing

Application backend tests belong under `App/Modules/*/tests/backend`. Frontend tests belong under `App/Modules/*/tests/frontend` when the module has frontend resources.

Add or update tests for observable application behavior, especially routing, commands, persistence, queues, scheduled work, mail, notifications, security, configuration, and integration boundaries.

## Documentation

When adding or changing application behavior, update the nearby docs or module README when the change affects how a developer uses, configures, tests, or operates the feature.

Keep application docs focused on the application. Do not copy framework-internal explanations unless they help the application developer make a correct decision.
